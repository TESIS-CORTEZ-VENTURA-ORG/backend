# HU-03-03/04/05/10/11/12 — Órdenes del POS (abrir mesa, tomar orden, modificadores, servido, anular, cuenta)

> **Épica:** E03 · **Sprint:** S2 · **Must** · **Estado:** 🟢 hecho (Incremento B — órdenes).

Dos modelos nuevos (RLS FORCE): **`orders`** (cuenta abierta sobre una mesa) y **`order_items`** (líneas con snapshot de nombre/precio/modificadores). Controller `@Controller('orders')` (`JwtAuthGuard` + `PoliciesGuard`). Subject CASL **`Order`** (staff `create`/`update`; manager/owner gestionan). Dinero = `Prisma.Decimal` devuelto como string `.toFixed(2)`. Todo el acceso vía `runInTenant` (tenant_id solo del JWT).

## HU-03-03 · Abrir mesa
```gherkin
WHEN el mesero abre una mesa libre (comensales)
THEN se crea una orden 'open' AND la mesa pasa a 'occupied' AND el mesero queda registrado
```
**Implementado ✅:** `POST /api/orders { tableId, guests?, idempotencyKey? }`. Valida que la mesa **exista** y esté **`free`** (si no → **409**); `guests ≤ table.capacity` (si no → **400**); `waiterId = claims.sub` (el mesero que abre). Crea `Order(status 'open')` y pone la mesa en **`occupied`**. **Idempotencia:** si se envía `idempotencyKey` y ya existe una orden con esa clave, se **devuelve la misma orden** (pre-check + catch P2002 en carrera) → reintentos seguros sin duplicar. El índice `@@unique([tenant_id, idempotency_key])` permite múltiples `NULL` (aperturas sin clave nunca colisionan). `@Audited('order.open')`.

## HU-03-04 · Tomar orden · HU-03-05 · Aplicar modificadores
```gherkin
WHEN agrega ítems a la orden con cantidad y modificadores
THEN cada ítem guarda nombre y precio del momento (snapshot) AND el precio incluye los modificadores
```
**Implementado ✅:** `POST /api/orders/:id/items { items: [{ menuItemId, qty, notes?, modifierIds? }] }`. Por cada ítem: carga el **`MenuItem`** (debe existir → si no **400**); resuelve `modifierIds` contra **`MenuModifier`** (cada uno debe pertenecer a ese plato → si no **400**); `unitPrice = menuItem.price + Σ modifier.priceDelta` (`Prisma.Decimal`); **snapshot** de `name` (plato) y de `modifiers` como JSONB `[{ name, priceDelta: number }]`. Crea `OrderItem(status 'pending')`. Devuelve la `OrderView` con `subtotal` recalculado. `@Audited('order.add_items')`.

## HU-03-10 · Marcar ítem servido (+ editar/quitar)
```gherkin
WHEN marca un ítem como servido THEN se registra la hora de servido (servedAt)
```
**Implementado ✅:** `PATCH /api/orders/:id/items/:itemId { qty?, status?, remove? }`. Si `remove: true` → **soft-delete** del ítem (`deleted_at`, sale del subtotal y de la vista). Si no, aplica `qty`/`status`; al fijar `status: 'served'` se sella **`servedAt = now()`** (con `sentToKitchenAt` presente —Inc C— habilita los KPIs de tiempo de servicio). `@Audited('order.update_item')`.

## HU-03-11 · Anular orden con razón
```gherkin
WHEN anula una orden con una razón THEN la orden pasa a 'void' AND la mesa se libera AND la razón queda registrada
```
**Implementado ✅:** `POST /api/orders/:id/void { reason }`. La **razón es obligatoria** (Zod `reason.min(1)` → falta = **400**). Solo se permite si el estado ∈ `{ open, sent_to_kitchen, served }` (si no → **409**). Pone `status = 'void'`, guarda `voidReason` y libera la mesa (`table.status = 'free'`). `@Audited('order.void')`.

## HU-03-12 · Solicitar cuenta
```gherkin
WHEN el mesero solicita la cuenta THEN la mesa se marca para cobro
```
**Implementado ✅ (sin endpoint nuevo):** "solicitar cuenta" reutiliza el endpoint existente del salón **`PATCH /api/tables/:id { status: 'bill' }`** (HU-03-01/02). No se añade ruta nueva: el estado `bill` ya forma parte del mapa POS y el cobro real es E04. El frontend BFF llama a `tables`, no a `orders`, para este paso.

## Contrato — vistas
- **OrderView:** `{ id, tableId, waiterId: string|null, guests, status, openedAt(ISO), items: OrderItemView[], subtotal: string(.toFixed2) }`.
- **OrderItemView:** `{ id, menuItemId, name, qty, unitPrice: string(.toFixed2), notes: string|null, modifiers: {name:string,priceDelta:number}[], status }`.
- `GET /api/orders/:id` devuelve la `OrderView` con sus ítems vivos (`deleted_at null`) y `subtotal = Σ unitPrice·qty`.

## RBAC
Subject **`Order`** (ya en `CaslAbilityFactory`): **staff** `can(['create','update'], 'Order')` → opera el POS (abrir mesa, tomar orden, servir, anular); manager/owner gestionan todo. Liberar/cambiar estado de mesa va por subject `Table` (staff `update`). `@Audited` en abrir / agregar ítems / editar ítem / anular.

## Multi-tenant
`orders` y `order_items` con `tenant_id NOT NULL`, RLS FORCE + policy `tenant_isolation` (`NULLIF(current_setting('app.tenant_id', true), '')::uuid`), verificado `relforcerowsecurity = t`. `order_items.kitchen_station_id` queda reservado (nullable, sin FK) para el KDS de Inc C.
